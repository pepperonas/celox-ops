# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

celox ops is a full-stack business management webapp for freelancers/IT consultants. German-language UI. **Multi-user with isolated workspaces** (per-user `owner_id` data isolation; admin-managed accounts, no public registration; JWT auth). Manages customers, orders, contracts, invoices (with PDF generation), leads, time tracking, expenses, the **Rainmaker** acquisition-activation module, and integrates with the Claude Token Tracker for AI usage transparency. The frontend uses a **Material Design 3 Expressive** dark theme.

```bash
docker compose up -d --build          # Build and start all services
docker compose up -d --build backend   # Rebuild backend only
docker compose up -d --build frontend  # Rebuild frontend only
docker compose up -d backend           # Recreate ohne Rebuild — nötig für .env-Änderungen! (restart übernimmt sie NICHT: Container behält seine Start-Umgebung)
```

## Architecture

### Backend (Python 3.12, FastAPI)
- **Entry point**: `backend/app/main.py` — FastAPI app with lifespan (creates tables + starts hourly cron)
- **Config**: `backend/app/config.py` — Pydantic Settings, all from `.env`
- **Database**: `backend/app/database.py` — async SQLAlchemy 2.0 with asyncpg, connection pooling
- **Auth**: `backend/app/auth.py` — JWT (python-jose), DB-backed `users` table, OAuth2PasswordBearer. `get_current_user` returns the `User` and sets the `current_owner_id` ContextVar (see Multi-tenancy). `require_admin` gates admin endpoints. `.env` admin is bootstrapped into a real DB user on first start. **Google-Login ("Sign in with Google")**: `POST /api/auth/google` verifiziert das GIS-ID-Token (google-auth: Signatur/exp/aud gegen `GOOGLE_CLIENT_ID` aus .env; `validate_google_claims` prüft zusätzlich iss/email_verified — DB-frei getestet) und meldet den Benutzer an, dessen `users.google_email` (unique, case-insensitiv verglichen, via manuellem `ALTER` migriert) zur E-Mail passt. Kein Auto-Signup (403 für unverknüpfte Konten); TOTP wird bei Google-Login bewusst nicht zusätzlich verlangt. Leere `GOOGLE_CLIENT_ID` ⇒ 503 + Button ausgeblendet (Login.tsx lädt das GIS-Script nur, wenn `GET /api/auth/info` eine `google_client_id` liefert — Laufzeit-Konfig, kein Frontend-Rebuild bei Änderung). JWKS-Fetch läuft via `asyncio.to_thread`; Rate-Limit 10/min/IP. Deps: `google-auth` + `requests`. Verknüpfung pflegt der Admin über `PATCH /api/users/{id}` (`google_email`; "" entfernt sie).
- **Multi-tenancy invariants (important)**: (1) **Public endpoints and background jobs** run with the ContextVar UNSET → queries are UNSCOPED (global). Any public route that returns owned data MUST resolve the user and `current_owner_id.set(user.id)` itself (see `routers/ical.py`, which authenticates via per-user `users.ical_token`). The cron (`run_cron`) sets the ContextVar per active user for the Rainmaker reminder; `check_overdue_invoices` is intentionally global. `backup.py` export sets it to `None` for a true full dump. (2) **Invoice numbers are unique PER OWNER** (`uq_invoice_owner_number`), not globally — `generate_invoice_number` is owner-scoped so each user has their own sequence. It takes a per-owner `pg_advisory_xact_lock` (held to commit) so concurrent creates (double-submit, or cron vs. the manual generate endpoint) serialize instead of colliding on the unique constraint; `flush_new_invoice` is a SAVEPOINT-based retry safety net. (3) When accepting an FK id (customer/order/contract/…) in a request body, validate it with a scoped `select` (returns None if it belongs to another tenant) before assigning — `with_loader_criteria` only hides rows on SELECT, it does not validate inserts.
- **Multi-tenancy / data isolation**: `backend/app/tenancy.py` — every owned model uses `OwnedMixin` (`owner_id` FK→users). A request-scoped `current_owner_id` ContextVar (set in `get_current_user`) drives two SQLAlchemy session events: `do_orm_execute` auto-filters every ORM SELECT (incl. column aggregates) to the owner via `with_loader_criteria`, and `before_flush` auto-stamps `owner_id` on new owned objects. Unset ContextVar (login, lifespan bootstrap, cron) ⇒ unscoped/global. **So routers generally need NO manual owner filtering** — but caches must be per-owner (e.g. dashboard `_stats_cache` is keyed by `owner_id`). `users`/`audit_log`/`document_templates` are global (not owned). Owned-model list is registered in `main.py` (`set_owned_models([...])` + `install_tenancy_events()`).
- **Pattern**: Each module has model → schema → router. All routers registered in main.py after app creation (to avoid circular imports).
- **Models inherit from `Base`** defined in `models/customer.py`. New models MUST import Base from there.
- **All list endpoints return paginated**: `{items: [], total, page, page_size, pages}` — NOT plain arrays. Exception: `GET /api/attachments` returns a plain list.
- **UUIDs**: All primary keys are UUID (asyncpg uses its own UUID type — serialize with `str()` for JSON).
- **Enum values**: Stay ASCII in code (`ueberfaellig`, `gekuendigt`, `halbjaehrlich`). German umlauts only in user-facing text/labels.
- **PDF generation**: WeasyPrint + Jinja2 templates in `backend/app/templates/`. Signature loaded as base64. Footer uses `@page @bottom-center` (not position:fixed).
- **Cron**: Background asyncio task in lifespan, hourly. Globally: marks overdue invoices (`check_overdue_invoices`). Then **per active user** (ContextVar scoped): Rainmaker daily reminder + **auto-recurring invoices** (`services/invoice_service.generate_due_recurring` — creates draft invoices for due contract billings; idempotent via `last_invoiced_date`, same logic as `POST /api/invoices/generate-recurring`).
- **Backup export** (`routers/backup.py`): Auto-discovers all tables via `Base.registry.mappers` — no manual list needed for new tables.

Alle 31 nativen Selects wurden migriert — **Stand: 0 `<select>` in `src/`**. Zusätzlich `:root { color-scheme: dark }` gesetzt, damit native Widgets (Datums-Picker/Kalender-Icon/Autofill) dunkel rendern.
- **Modals MUST portal to `document.body`** (`createPortal`): the route wrapper `.page-enter` (Layout.tsx) animates `transform`, and a transformed ancestor becomes the containing block for `position: fixed` descendants — a modal rendered inside a page darkens its overlay but mis-positions/clips the panel (looked "invisible until refresh"). `LinkedInImportModal` + `LeadDiscoveryModal` render via `createPortal(..., document.body)`; new modals opened from inside a page must do the same. **Gilt auch für den `Fab`** (`components/Fab.tsx`): er ist `position: fixed`, wurde aber im Seiten-Body gerendert → ankerte an der Inhalts-Unterkante (bei kurzen Seiten mitten im Content statt unten rechts am Viewport). Jetzt portalt der `Fab` selbst an `document.body` — alle `<Fab/>`-Nutzungen sind damit fix. Jedes neue seiteninterne `position: fixed`-Element braucht denselben Portal.
- **Motion / navigation craft** (`utils/transitions.ts`): in-app navigation runs through `useAppNavigate()` (drop-in for `useNavigate`), which sets a direction flag (`html[data-nav="forward"|"back"]`, pass `{ back: true }` for up-a-level) and then navigates. The incoming content reveals via a **GPU-only CSS animation** (`.page-enter` → `page-in-fwd`/`page-in-back`, opacity + transform) on a route-keyed wrapper in `Layout`. **Important:** this deliberately does NOT use the View Transitions API — VT snapshots the destination, which flickers/janks with our async-loading, Chart.js-canvas-heavy views (it captures the loading state, then hard-swaps when data/canvas arrive). The cheap reveal animates the real DOM, so async content just appears inside the fading container. `useScrollRestoration(mainRef)` remembers the `<main>` scrollTop per history entry and restores it on POP (no animated scroll); it also sets `html[data-pop="1"]` so entrance animations don't replay on back/forward. To avoid double motion, `.page-enter .md-stagger > *` is disabled (the page reveal is the single entrance). Reactive `TiltCard` (Dashboard KPIs) tilts toward the cursor, gated to `(hover:hover) and (pointer:fine)` + reduced-motion. The Rainmaker "Erledigt" moment uses `.rm-complete-exit` (queue item leaves) + `.rm-ring-pop` (progress ring). Reduced motion → instant navigation. The scroll container is `<main>`, NOT the window.

- **Action-first acquisition tool** — surfaces *what to do today*, not a contact list. Backend under `/api/rainmaker` (`routers/rainmaker.py`), engine helpers in `services/rainmaker_service.py`. Frontend pages in `pages/rainmaker/`.
- **5 models** (`rainmaker_lead`, `rainmaker_activity`, `rainmaker_settings`, `rainmaker_streak`, `rainmaker_template`) — registered for `create_all` in `main.py`, `native_enum=False` enums, **owner-scoped** (`OwnedMixin`); `settings`/`streak` are now per-user single-row tables (`get_or_create_*` returns the current owner's row via auto-scoping).
- **Activation engine**: a lead's "next action" = earliest planned activity by `due_date`. An active lead (status not `won`/`lost`/`dormant`) without a planned activity is **"rotting"** and surfaced in red. `GET /today` returns the queue (planned, due ≤ today, sorted by priority + overdueness) + rotting list + progress.
- **Offenes To-do zählt als nächster Schritt (2026-07):** `is_rotting` schaut nur auf Rainmaker-Aktivitäten — ein Lead mit einem offenen **To-do** (aber ohne geplante Aktivität) galt fälschlich als verrottend. Jetzt reicht `lead_response(lead, has_open_todo)` das durch (`needs_next_action = is_rotting(lead) and not has_open_todo`); `get_lead`/`list_leads` liefern per `_leads_with_open_todos` (eine Batch-Abfrage, owner-scoped) mit, welche Leads offene To-dos haben. Wirkt auf **beide** Stellen (Pipeline-„rotting"-Rand + LeadDetail-Banner, beide nutzen `needs_next_action`); im Banner erscheint das frühestfällige offene To-do als nächster Schritt. Tests: `test_rainmaker.py`.
- **Next-Action-Zwang**: `POST /activities/{id}/complete` logs done (+outcome/notes) and **atomically requires** a next planned action — unless the lead is closed (`won`/`lost`/`dormant`). Returns 400 otherwise. After mutating it reloads `lead.activities` (`refresh(..., attribute_names=["activities"])`) so the recomputed next-action isn't stale.

## Key Gotchas
- **Download-Dateinamen sind zentralisiert (2026-07)**: ALLE Downloads/Mail-Anhänge benennen über `services/filenames.py` — `download_name(*teile, ext=)` baut `Typ_Kunde_Kennung_Datum.ext` (Teile einzeln geslugt via `slug_name`: Umlaute→ae/oe/ue/ss, nur ASCII `[A-Za-z0-9.-]`, header-sicher — ein rohes `"` im Firmennamen brach früher den Content-Disposition-Header in documents.py); `customer_label(customer)` = company‖name. Beispiele: `Rechnung_Beispiel-GmbH_CO-2026-0001.pdf`, `Angebot_…`, `Zahlungserinnerung_…`, `Stundennachweis_…_{von}_{bis}.pdf`, `Vertragsdokumente_….zip`, `DSGVO-Export_…_{datum}.json`, `PageSpeed_{domain}_…`, `Monatsbericht_2026-07.pdf`, `EUeR_2026.csv`. `send_email(..., attachment_name=)` überschreibt den internen Speichernamen im Mail-Anhang (Speicherpfade/`{invoice_number}.pdf` blieben unverändert; nur der Angebots-SPEICHERname ist jetzt auch geslugt — ein `/` im Auftragstitel brach den Pfad). Frontend liest den Namen aus dem Header via `utils/downloadName.ts::filenameFromDisposition` (inkl. RFC-5987 `filename*=`) statt eigene Namen zu erfinden — `api/invoices.ts` `downloadPdf`/`downloadReminderPdf` geben `{blob, filename}` zurück. Neue Downloads: IMMER `download_name` + Header-Auslesen nutzen. Tests: `test_filenames.py` + `downloadName.test.ts`.
- **Pydantic `date` field shadowing**: If a model has a field named `date`, import as `from datetime import date as DateType` to avoid `date | None` failing in subsequent classes in the same file.
- **Docker .env `$` escaping**: bcrypt hashes in `ADMIN_PASSWORD_HASH` must escape `$` as `$$`.
- **page_size limit**: Set to `le=1000` on all list endpoints (frontend requests 1000 for dropdowns).
- **PDF footer**: Use `@page @bottom-center` in CSS, NOT `position: fixed` (causes overlap with content).
- **PDF Anschriftenblock**: the customer address field is single-line free text — `services/address_format.py::format_address_lines` splits it into DIN-5008-style lines for invoice/quote/reminder PDFs (comma-separated, comma-less via 5-digit-PLZ boundary, multiline passthrough; hyphens in streets/cities/number ranges stay intact; a trailing standalone "Deutschland" is dropped for domestic mail). Registered as Jinja filter `address_lines` on all three pdf_service Environments — new templates must use it instead of dumping `customer.address` raw.
- **Nginx timeout**: Set to 120s in `nginx/default.conf` for PageSpeed and other long-running API calls. Max body size 10M.
- **Nginx 502 after deploy — dauerhaft gefixt (2026-07):** Der Docker-nginx (`nginx/default.conf`) hat jetzt `resolver 127.0.0.11` (Dockers DNS) + **variable `proxy_pass`** (`set $backend_up backend; proxy_pass http://$backend_up:8000;`). Dadurch löst nginx die Upstream-Container **pro Request** neu auf, statt die IP beim Start zu cachen — ein neu gebauter Container (neue IP) wird sofort gefunden, **ohne nginx-Restart**. Früher zeigte nginx nach jedem Rebuild auf die tote alte IP → 502-Fenster (fehlgeschlagene Lazy-Chunk-Imports im offenen Tab). **Wichtig:** variable `proxy_pass` OHNE URI-Teil leitet die Original-URI unverändert weiter (Location-Präfix == Upstream-Präfix, daher kein Rewrite nötig). **Bind-Mount-Inode-Bug:** `nginx/default.conf` ist als einzelne Datei gemountet → `git pull` ersetzt den Inode, `docker compose restart nginx` übernimmt die neue Config NICHT (alter Inode bleibt gemountet); nur `docker compose up -d --force-recreate nginx` bindet die neue Datei ein. `scripts/auto-deploy.sh` macht das jetzt automatisch bei `nginx/`-Änderungen.
- **ZWEI nginx-Ebenen (wichtig für Timeouts):** (1) der **öffentliche System-nginx** des VPS (`/etc/nginx/sites-available/ops.celox.io`, NICHT im Repo, Certbot-TLS) proxyt `ops.celox.io` → `127.0.0.1:8090`; (2) der **Docker-nginx** (`nginx/default.conf`, im Repo) → `backend:8000`. Ein langes Request (KI-Web-Suche) muss auf **beiden** Ebenen ein hohes `proxy_read_timeout` haben, sonst 504 obwohl das Backend fertig wird. Beide haben eine dedizierte Location `/api/rainmaker/discover/ai/` mit **300s** (Default sonst 60s außen / 180s innen). Ändert sich das äußere vhost, per SSH editieren + `nginx -t && systemctl reload nginx` (Backup anlegen). Der Anthropic-Client hat zusätzlich 150s/Call.
- **Signature in documents**: Image goes ABOVE the line (`border-top`), text BELOW. Signature HTML is a placeholder `{signature_html}` replaced during rendering.
- **Discount storage**: Stored as `discount_type` (percent/fixed), `discount_value`, `discount_reason` on Invoice model — NOT as negative positions. Guard: `discount_value is not None and discount_value != 0`.
- **Discount clear on PUT**: `update_invoice` distinguishes "field present in body" (`"discount_type" in update_data`) from `is not None` — the form sends explicit `null` to remove a discount, and a `None`-check would recompute totals with the OLD discount while the row is nulled (silent money mismatch, also leaked into the regenerated PDF). Fixed 2026-07.
- **Special terms**: Stored as JSON array string in `special_terms` field. Single string also supported (backward compatible).
- **Storno = Gutschrift (`POST /invoices/{id}/credit-note`, überarbeitet 2026-07)**: Eine gestellte Rechnung wird nie inhaltlich „zurückgesetzt" — sie wird durch eine **Gutschrift** storniert. Die erzeugt einen Datensatz mit **eigenem Nummernkreis `GS-YYYY-NNNN`** (`generate_credit_note_number`, per Advisory Lock serialisiert wie die Rechnungsnummer), gespiegelten Beträgen (`build_credit_note_positions`: `-abs` über Decimal, `menge`/Text bleiben), `is_credit_note=True`, `credit_note_for` (FK) **+ `credit_note_for_number`** (denormalisiert für PDF/Anzeige, Migration `scripts/add_credit_note_number.sql`). **Die Rechnungsnummer des Originals bleibt immer bestehen** und wird nie neu vergeben. **Status-Logik (`credit_note_statuses`, entscheidend für korrekte Zahlen):** Original `bezahlt` → beide bleiben `bezahlt` (Netting: +X/−X, Rückfluss im Monat der Gutschrift); Original `gestellt`/`ueberfaellig` → **beide `storniert`** (Neutralisierung — `storniert` zählt weder in der EÜR (nur `bezahlt`) noch in offenen Forderungen). **Frühere Bugs (gefixt):** GS bekam hart `bezahlt` → negativer Phantom-Umsatz in der EÜR bei nie bezahlten Rechnungen, Original blieb offene Forderung; `tax_exempt` wurde nicht kopiert → Kleinunternehmer-Gutschrift wies 19 % USt aus; kein PDF; keine Nummern-Serialisierung. PDF wird jetzt sofort erzeugt, das Template rendert im Gutschrift-Modus (Kopf „Gutschriftnummer/-datum" + Pflichtzeile „Storniert Rechnung …" nach §14 Abs. 4 UStG, Zahlungsblock/IBAN → Erstattungshinweis). Korrigierte Fassung danach über **„Duplizieren"** (neuer Entwurf, neue Nummer). UI: Button „Stornieren (Gutschrift)" + **Bestätigungsdialog** (createPortal, erklärt die Statuswirkung je nach Ausgangslage; kein Undo — Nummer ist vergeben), gegenseitige Verlinkung Original ↔ Gutschrift (`credit_note_id`/`credit_note_number` füllt nur `get_invoice`). Tests: `test_credit_note.py`.
- **Async httpx in refresh-drafts**: Uses `httpx.AsyncClient` (not sync `httpx.get`) to avoid blocking the event loop.
- **PDF generation in threads**: All `generate_*_pdf()` calls wrapped via `asyncio.to_thread()` since WeasyPrint is sync and would block the event loop for 2-5s per render.
- **PDF regen on invoice save is backgrounded**: `update_invoice` (PUT) used to regenerate the PDF synchronously in-request — WeasyPrint (2-5s) + token-tracker fetch (`timeout` now 8s, was 30s) + GitHub commit fetch (N+1) made saving take 4-8s. It now schedules `_regenerate_pdf_bg(invoice.id)` via FastAPI `BackgroundTasks` (own session); the PUT returns in ~130ms and the PDF updates a moment later (brief staleness window if the user downloads immediately). The explicit `POST /{id}/generate-pdf` stays synchronous (user waits for the file). Create never generated a PDF (already fast).
- **CORS_ORIGINS env var required**: Empty value blocks all cross-origin requests. Set to `https://ops.celox.io` (or local dev origin) in `.env`.

### Legal-Compliance-Tracking (Rechtsdokumente)
Stellt sicher, dass jeder Kunde die Pflicht-Rechtsdokumente (Default **AGB + AVV**) unterschrieben vorliegen hat. Router `routers/compliance.py` (`/api/compliance/*`): `GET /overview` (alle Kunden × Pflichtdocs, Lücken zuerst), `GET /customer/{id}`, `POST /mark` (manuell ab-/anhaken), `POST /upload` (unterschriebenes PDF → legt einen `attachments`-Eintrag an **und** den Compliance-Record an), `PUT /templates/{id}/required` (Pflicht-Flag toggeln).
- **Pflicht-Set ist konfigurierbar** über die neue Spalte `document_templates.compliance_required` (nullable: `NULL`=nie konfiguriert → Seed setzt Default für AGB+AVV; `true/false`=Nutzerwahl bleibt bei Re-Seed erhalten). **Spalte via manuellem `ALTER` auf bestehende Tabelle** (`ALTER TABLE document_templates ADD COLUMN compliance_required boolean;`), danach Seed nachziehen.
- **`compliance_records`** (auto-erstellt via `create_all`): eine Zeile pro (Kunde, Vorlage), unique. `signed_at` gesetzt = erfüllt; `method` = `upload`|`manual`; `attachment_id` verweist auf die hochgeladene Datei. „Erfüllt" wird daraus berechnet — Anhänge bleiben die Single Source of Truth für die Datei.
- **Frontend**: Nav-Seite **Rechtsdokumente** (`/rechtsdokumente`, `pages/Compliance.tsx`) mit Summary + Pro-Kunde-Karten (PDF erstellen / Hochladen / Abhaken) + Pflicht-Konfiguration. Im Kundendetail: Header-Badge „Legal" + `CustomerComplianceCard` im Dokumente-Tab. PDF-Erstellung nutzt die bestehenden `routers/documents.py`-Endpoints.
