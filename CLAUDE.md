# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

celox ops is a full-stack business management webapp for freelancers/IT consultants. German-language UI. **Multi-user with isolated workspaces** (per-user `owner_id` data isolation; admin-managed accounts, no public registration; JWT auth). Manages customers, orders, contracts, invoices (with PDF generation), leads, time tracking, expenses, the **Rainmaker** acquisition-activation module, and integrates with the Claude Token Tracker for AI usage transparency. The frontend uses a **Material Design 3 Expressive** dark theme.

## Commands

### Production
```bash
docker compose up -d --build          # Build and start all services
docker compose up -d --build backend   # Rebuild backend only
docker compose up -d --build frontend  # Rebuild frontend only
docker compose up -d backend           # Recreate ohne Rebuild — nötig für .env-Änderungen! (restart übernimmt sie NICHT: Container behält seine Start-Umgebung)
docker compose down                    # Stop all
```

```bash
# Frontend: http://localhost:5173 (Vite HMR)
```


### Tests
```bash

cd backend && python -m pytest -q          # locally needs the deps; otherwise run in the container:

  -v "$PWD/backend/tests:/app/tests" backend python -m pytest -q

```
CI runs both suites on every push (`.github/workflows/ci.yml`): backend = ruff + router-import + `pytest`; frontend = `tsc` + `vitest` + build.

```bash
docker exec celox-ops-db-1 psql -U celoxops -d celoxops -c "ALTER TABLE tablename ADD COLUMN colname TYPE;"
```

### Deployment to VPS
```bash
tar czf /tmp/celox-ops.tar.gz --exclude='.git' --exclude='node_modules' --exclude='.DS_Store' --exclude='._*' --exclude='.env' --exclude='.claude' .
scp /tmp/celox-ops.tar.gz root@YOUR_VPS:/tmp/
ssh root@YOUR_VPS 'cd /opt/celox-ops && tar xzf /tmp/celox-ops.tar.gz && rm /tmp/celox-ops.tar.gz && find . -name "._*" -delete && docker compose up -d --build backend frontend && docker compose restart nginx'
```
**Important**: Always `restart nginx` after rebuilding backend/frontend — Docker assigns new container IPs and nginx caches the old DNS resolution, causing 502 errors.

## Architecture

### Backend (Python 3.12, FastAPI)
- **Entry point**: `backend/app/main.py` — FastAPI app with lifespan (creates tables + starts hourly cron)
- **Config**: `backend/app/config.py` — Pydantic Settings, all from `.env`
- **Database**: `backend/app/database.py` — async SQLAlchemy 2.0 with asyncpg, connection pooling
- **Auth**: `backend/app/auth.py` — JWT (python-jose), DB-backed `users` table, OAuth2PasswordBearer. `get_current_user` returns the `User` and sets the `current_owner_id` ContextVar (see Multi-tenancy). `require_admin` gates admin endpoints. `.env` admin is bootstrapped into a real DB user on first start. **Google-Login ("Sign in with Google")**: `POST /api/auth/google` verifiziert das GIS-ID-Token (google-auth: Signatur/exp/aud gegen `GOOGLE_CLIENT_ID` aus .env; `validate_google_claims` prüft zusätzlich iss/email_verified — DB-frei getestet) und meldet den Benutzer an, dessen `users.google_email` (unique, case-insensitiv verglichen, via manuellem `ALTER` migriert) zur E-Mail passt. Kein Auto-Signup (403 für unverknüpfte Konten); TOTP wird bei Google-Login bewusst nicht zusätzlich verlangt. Leere `GOOGLE_CLIENT_ID` ⇒ 503 + Button ausgeblendet (Login.tsx lädt das GIS-Script nur, wenn `GET /api/auth/info` eine `google_client_id` liefert — Laufzeit-Konfig, kein Frontend-Rebuild bei Änderung). JWKS-Fetch läuft via `asyncio.to_thread`; Rate-Limit 10/min/IP. Deps: `google-auth` + `requests`. Verknüpfung pflegt der Admin über `PATCH /api/users/{id}` (`google_email`; "" entfernt sie).
- **Multi-tenancy invariants (important)**: (1) **Public endpoints and background jobs** run with the ContextVar UNSET → queries are UNSCOPED (global). Any public route that returns owned data MUST resolve the user and `current_owner_id.set(user.id)` itself (see `routers/ical.py`, which authenticates via per-user `users.ical_token`). The cron (`run_cron`) sets the ContextVar per active user for the Rainmaker reminder; `check_overdue_invoices` is intentionally global. `backup.py` export sets it to `None` for a true full dump. (2) **Invoice numbers are unique PER OWNER** (`uq_invoice_owner_number`), not globally — `generate_invoice_number` is owner-scoped so each user has their own sequence. It takes a per-owner `pg_advisory_xact_lock` (held to commit) so concurrent creates (double-submit, or cron vs. the manual generate endpoint) serialize instead of colliding on the unique constraint; `flush_new_invoice` is a SAVEPOINT-based retry safety net. (3) When accepting an FK id (customer/order/contract/…) in a request body, validate it with a scoped `select` (returns None if it belongs to another tenant) before assigning — `with_loader_criteria` only hides rows on SELECT, it does not validate inserts.
- **Multi-tenancy / data isolation**: `backend/app/tenancy.py` — every owned model uses `OwnedMixin` (`owner_id` FK→users). A request-scoped `current_owner_id` ContextVar (set in `get_current_user`) drives two SQLAlchemy session events: `do_orm_execute` auto-filters every ORM SELECT (incl. column aggregates) to the owner via `with_loader_criteria`, and `before_flush` auto-stamps `owner_id` on new owned objects. Unset ContextVar (login, lifespan bootstrap, cron) ⇒ unscoped/global. **So routers generally need NO manual owner filtering** — but caches must be per-owner (e.g. dashboard `_stats_cache` is keyed by `owner_id`). `users`/`audit_log`/`document_templates` are global (not owned). Owned-model list is registered in `main.py` (`set_owned_models([...])` + `install_tenancy_events()`).
- **Pattern**: Each module has model → schema → router. All routers registered in main.py after app creation (to avoid circular imports).
- **Models inherit from `Base`** defined in `models/customer.py`. New models MUST import Base from there.
- **All list endpoints return paginated**: `{items: [], total, page, page_size, pages}` — NOT plain arrays. Exception: `GET /api/attachments` returns a plain list.
- **UUIDs**: All primary keys are UUID (asyncpg uses its own UUID type — serialize with `str()` for JSON).
- **Enum values**: Stay ASCII in code (`ueberfaellig`, `gekuendigt`, `halbjaehrlich`). German umlauts only in user-facing text/labels.
- **PDF generation**: WeasyPrint + Jinja2 templates in `backend/app/templates/`. Signature loaded as base64. Footer uses `@page @bottom-center` (not position:fixed).
- **Cron**: Background asyncio task in lifespan, hourly. Globally: marks overdue invoices (`check_overdue_invoices`). Then **per active user** (ContextVar scoped): Rainmaker daily reminder + **auto-recurring invoices** (`services/invoice_service.generate_due_recurring` — creates draft invoices for due contract billings; idempotent via `last_invoiced_date`, same logic as `POST /api/invoices/generate-recurring`).
- **Backup export** (`routers/backup.py`): Auto-discovers all tables via `Base.registry.mappers` — no manual list needed for new tables.

### Frontend (React 18, TypeScript, TailwindCSS, Vite)
- **Entry**: `frontend/src/main.tsx` → `App.tsx` (routing) → `Layout.tsx` (sidebar + content)
- **Responsive shell**: the nav is a persistent rail at `md+` (collapsible) and an off-canvas **drawer below `md`** (hamburger in the top bar opens it, scrim closes it, route change auto-closes it). Content is full-width on phones. Safe-area insets applied to the header (top), `<main>` (bottom) and the FAB. Detail-page header button rows use `flex-wrap`; the Rainmaker `ProgressHeader` stacks (ring+label / streak+points) below `sm`. Verified at 390px: no horizontal overflow on any page.
- **State**: Zustand for auth (`store/authStore.ts`), localStorage for JWT
- **API client**: `api/client.ts` — Axios with JWT interceptor, 401 redirect
- **Types**: All in `frontend/src/types/index.ts` — must match backend Pydantic schemas exactly
- **Routing**: German paths (`/kunden`, `/auftraege`, `/rechnungen`, `/vorgemerkt`, `/ausgaben`, etc.)
- **Theme**: **Material Design 3 Expressive** (dark). The token layer lives in `index.css`: MD3 tonal surfaces as RGB-channel CSS vars (`--c-bg`, `--c-surface`, `--c-accent`, …) so Tailwind opacity modifiers (`bg-danger/10`) keep working; expressive shape scale, elevation (`shadow-elev-1..3`), motion tokens (easing `ease-emphasized/decelerate/spring`, durations `duration-short/medium/long`), state-layer helper `.md-state`, pill `.btn-*` with shape-morph on press, `.fab`, `.md-spinner`, keyframe utilities (`animate-md-enter/scale/pop`, `.md-stagger`, `.md-skeleton`). `tailwind.config.ts` maps the legacy semantic names (`bg`, `surface`, `accent`, …) onto these vars. Reusable MD3 components: `PageHeader`, `Fab`, `FilterChips`, `SegmentedButtons`, `LoadingIndicator` (plus restyled `StatusBadge`, `DataTable`, dialogs). Labels are sentence-case (no uppercase micro-labels). `prefers-reduced-motion` disables animations.
- **Modals MUST portal to `document.body`** (`createPortal`): the route wrapper `.page-enter` (Layout.tsx) animates `transform`, and a transformed ancestor becomes the containing block for `position: fixed` descendants — a modal rendered inside a page darkens its overlay but mis-positions/clips the panel (looked "invisible until refresh"). `LinkedInImportModal` + `LeadDiscoveryModal` render via `createPortal(..., document.body)`; new modals opened from inside a page must do the same.
- **Motion / navigation craft** (`utils/transitions.ts`): in-app navigation runs through `useAppNavigate()` (drop-in for `useNavigate`), which sets a direction flag (`html[data-nav="forward"|"back"]`, pass `{ back: true }` for up-a-level) and then navigates. The incoming content reveals via a **GPU-only CSS animation** (`.page-enter` → `page-in-fwd`/`page-in-back`, opacity + transform) on a route-keyed wrapper in `Layout`. **Important:** this deliberately does NOT use the View Transitions API — VT snapshots the destination, which flickers/janks with our async-loading, Chart.js-canvas-heavy views (it captures the loading state, then hard-swaps when data/canvas arrive). The cheap reveal animates the real DOM, so async content just appears inside the fading container. `useScrollRestoration(mainRef)` remembers the `<main>` scrollTop per history entry and restores it on POP (no animated scroll); it also sets `html[data-pop="1"]` so entrance animations don't replay on back/forward. To avoid double motion, `.page-enter .md-stagger > *` is disabled (the page reveal is the single entrance). Reactive `TiltCard` (Dashboard KPIs) tilts toward the cursor, gated to `(hover:hover) and (pointer:fine)` + reduced-motion. The Rainmaker "Erledigt" moment uses `.rm-complete-exit` (queue item leaves) + `.rm-ring-pop` (progress ring). Reduced motion → instant navigation. The scroll container is `<main>`, NOT the window.
- **Charts**: Chart.js + react-chartjs-2. Cast options as `any` to avoid TS issues with Chart.js types.
- **Tab persistence**: CustomerDetail uses URL hash (`#auftraege`, `#dokumente`, `#tokens`) for tab state.

### Rainmaker (Acquisition Activation)
- **Action-first acquisition tool** — surfaces *what to do today*, not a contact list. Backend under `/api/rainmaker` (`routers/rainmaker.py`), engine helpers in `services/rainmaker_service.py`. Frontend under `/rainmaker/*` (pages in `pages/rainmaker/`), own pill sub-nav (`RainmakerNav`) + footer.
- **5 models** (`rainmaker_lead`, `rainmaker_activity`, `rainmaker_settings`, `rainmaker_streak`, `rainmaker_template`) — registered for `create_all` in `main.py`, `native_enum=False` enums, **owner-scoped** (`OwnedMixin`); `settings`/`streak` are now per-user single-row tables (`get_or_create_*` returns the current owner's row via auto-scoping).
- **Activation engine**: a lead's "next action" = earliest planned activity by `due_date`. An active lead (status not `won`/`lost`/`dormant`) without a planned activity is **"rotting"** and surfaced in red. `GET /today` returns the queue (planned, due ≤ today, sorted by priority + overdueness) + rotting list + progress.
- **Next-Action-Zwang**: `POST /activities/{id}/complete` logs done (+outcome/notes) and **atomically requires** a next planned action — unless the lead is closed (`won`/`lost`/`dormant`). Returns 400 otherwise. After mutating it reloads `lead.activities` (`refresh(..., attribute_names=["activities"])`) so the recomputed next-action isn't stale.
- **Gamification**: daily quota (settings) vs `count_done_today`; points per type (call 10 · visit 20 · email/message/follow_up 5; ×1.5 at streak ≥ 7) via `register_completion`. **Working-day streak**: only Mon–Fri count (weekends are bonus = points only, never break it); missed working days consume a monthly **freeze** budget (`rainmaker_streak.freeze_remaining`, replenished to `settings.freezes_per_month` via `_ensure_monthly_freezes`) before the streak resets. `display_streak`/`get_streak_display` compute the live value (0 if missed working days exceed remaining freezes). The `freeze_remaining`/`freeze_period`/`freezes_per_month` columns were added via `ALTER TABLE` (create_all doesn't backfill columns on existing tables).
- **Reminder**: `check_rainmaker_reminder(db, user)` runs inside the hourly `run_cron` loop (NOT APScheduler), **once per active user** (ContextVar scoped, so settings/streak/quota are that user's). Sends one mail/day at `reminder_time` to `user.email` (fallback `BUSINESS_EMAIL`) when the quota is unmet; per-owner in-memory dedupe (`_reminder_sent_on` is a `{user_id: date}` dict). Email channel only; no-ops without SMTP.
- **Templates** (`/templates`): email/message with `{company}`/`{contact_name}`/`{role}` placeholders; LeadDetail can launch a `mailto:` from a template with placeholders substituted.
- **Acquisition goals** (`rainmaker_goal`, `/goals` CRUD + `/goals/seed` idempotent): user-defined channels (e.g. "Neukunden Telefon-Akquise", "LinkedIn anschreiben") with a `suggested_type` (pre-fills the action type when planning, overridable) and a **daily_target**. Activities optionally carry `goal_id` (column added via `ALTER`; the `rainmaker_goal` table itself is auto-created by create_all). `/today` returns per-goal `done_today/daily_target` (shown as "Tagesziele" bars) plus `total_leads`. The Heute empty-state is context-aware via `total_leads`/`done_today` (no leads → "leg den ersten an"; nothing due → "heute nichts fällig"; only truly cleared → "🎉").
- **Traumziel (dream goal)** — expected-value motivation engine (`/rainmaker/traumziel`, `GET /api/rainmaker/dream`): every completed acquisition action carries statistical € toward a dream object ("ein Nein am Telefon ist trotzdem 225 € Richtung Porsche"). Assumptions live as 8 `dream_*` columns on `rainmaker_settings` (goal key/name/price, savings-rate %, Ø deal value, contacts-per-win, start date, mode ev|invoices — **added via manual `ALTER TABLE` on existing DBs**). EV engine in `rainmaker_service.py`: `DREAM_EV_WEIGHTS` (call 1.0 · visit 2.5 · email/message 0.4 · follow_up 0.8 · note 0), `dream_ev_per_contact` (deal × rate / contacts), `dream_activities_ev`, `dream_projected_date` (28-day pace, None beyond 50 y). `/dream` aggregates done-activity counts since `dream_start_date` (auto-set to today on first visit), won leads (updated_at proxy) as "realized", paid invoices (for `invoices` mode = später, wenn das Business läuft). Frontend `pages/rainmaker/DreamGoal.tsx` + `dreamPresets.ts` (researched presets: Cayenne Turbo Electric 165.5k · Brabus Bodo 1.2M · Taycan Turbo GT · 911 Turbo S · G 63 · custom): hero card, road viz (1.000 € = 1 km, milestones 📋🛞🎨🪑🔑), randomized scenario cards (🎲 shuffle), what-if slider (months-earlier at current pace), inline config (PUT /settings).
- **LinkedIn-Import** (Pipeline header button → `LinkedInImportModal`): imports the LinkedIn **official free GDPR data export** — either the **complete export ZIP** (preferred; unpacked IN-MEMORY server-side with zip-bomb guards) or a single Connections.csv. From the ZIP, `services/linkedin_import.py` additionally parses `Invitations.csv` (**pending outgoing invites** not yet in Connections → candidates with status `contacted`, own preview section, deselected by default; **confirmed connections → status `connected`** — eigene Pipeline-Stufe „Vernetzt" zwischen contacted und in_conversation) and `messages.csv` (grouped per partner URL; own profile detected as most-frequent sender → matching leads get status `in_conversation` and their messages imported as **done `message` activities** with historical `completed_at`, deliberately WITHOUT `register_completion` so points/streak stay clean; capped `MAX_MESSAGES_PER_LEAD=20`, 300-char snippets). Connections parser handles the "Notes:" preamble, EN+DE headers, BOM, quoted commas; contacts without a company get the person name as `company` (NOT NULL). Endpoints: `POST /api/rainmaker/import/linkedin/preview` (multipart ZIP/CSV → rows with `source`/`status`/`message_count`/`messages` + `duplicate` flags) and `POST /api/rainmaker/import/linkedin` (selected rows → leads incl. status + activities; re-checks duplicates server-side). Dedup is **per owner** (tenancy-scoped) by normalized profile URL or contact name, plus in-file dupes — and duplicates are **ENRICHED instead of skipped**: status upgraded along new→contacted→connected→in_conversation (never past negotiation/closed stages; eine offene Anfrage kann einen bereits vernetzten Lead nie zurückstufen), missing email/role/website filled, message history attached if the lead has no `[LinkedIn`-prefixed activities yet (idempotent re-upload); the preview pre-selects duplicates that carry messages ('wird angereichert'). Mapping: `source="LinkedIn-Import"`, `tags=["linkedin"]`, `website`=profile URL, notes= "LinkedIn-Kontakt seit <Connected On>" bzw. "Kontaktanfrage gesendet am <Sent At> (noch nicht angenommen)". Limits: 10 MB upload / 20k rows. Modal: **das ganze Dialog-Panel** ist Dropzone (ZIP+CSV per Drag&Drop, Accent-Ring beim Ziehen; ein window-weiter dragover/drop-Guard verhindert, dass ein daneben abgelegtes ZIP vom Browser geöffnet wird), source filter chips, status column with 💬 message badge. **Pipeline board fetches ALL lead pages** (the API caps `page_size` at 1000 — with >1000 leads a single page showed wrong columns/zero counts) and render-caps each column at 100 cards with a '+N weitere anzeigen' button. **LeadDetail** offers a compliant LinkedIn messaging flow (no public messaging API — partner-only, automation violates ToS): 'LinkedIn'-button opens the profile; a 'LinkedIn-Vorlage…' select (templates with channel `message`) copies the placeholder-filled text to the clipboard and opens the profile to paste.
- **Google Places key in settings**: der Places-API-Key liegt **pro Workspace** in `app_settings` (`google_places_api_key`, via manuellem ALTER migriert; env `GOOGLE_PLACES_API_KEY` bleibt globaler Fallback). Settings-Router gibt ihn NIE im Klartext zurück (`google_places_configured` + maskierter `google_places_key_hint` via `services/places_usage.mask_key`). **Eigener Nutzungszähler** (`google_places_calls`/`_period`, Monats-Reset via `places_usage.bump`/`calls_this_month`) — Google gibt das Restkontingent per Key nicht her; die Settings-UI zeigt den eigenen Zähler + Link zur echten Google-Quota/Abrechnung. `discover_google` meldet REQUEST_DENIED/OVER_QUERY_LIMIT als 422 (Zähler steigt dann NICHT). Settings.tsx: Sektion mit Key-Eingabe (password, maskiert), Entfernen, Nutzung, aufklappbare Anleitung.
- **Lead-Discovery** (Pipeline header „Leads finden" → `LeadDiscoveryModal`): automatische Firmensuche nach Branche + Ort (Batch: mehrere Branchen x mehrere Staedte in einem Durchlauf mit Live-Fortschritt pro Kombination; Cross-Kombi-Dedup + `sortByQuality` (E-Mail-Leads zuerst) + "nur mit E-Mail"-Toggle in discoveryBatch.ts, getestet). `services/lead_discovery.py` fragt **OpenStreetMap/Overpass** (kostenlos, kein Key; `SEGMENT_OSM_TAGS` mappt Segmente auf OSM-Tags, `build_overpass_query`/`parse_overpass` netzfrei getestet) oder **Google Places** (nur wenn `GOOGLE_PLACES_API_KEY` gesetzt, sonst 503). Endpoints `POST /api/rainmaker/discover/preview` (Kandidaten mit `duplicate`-Flags, owner-scoped) und `/discover/import` (Auswahl anlegen; source="OpenStreetMap"/"Google Places", tags=["discovery", segment], status new, **`email` wird mitgespeichert**).
  - **Datenqualität „keine Karteileichen" (2026-07):** Jeder zurückgegebene Kandidat MUSS eine **Website** haben, die **live erreichbar** ist (`website_alive`/`filter_live_websites`: HEAD→GET-Fallback, Redirects, <400=ok, 8er-Parallelität, 4 s Timeout — tote/geparkte Domains fliegen raus). Zusätzlich: **OSM verlangt Website UND E-Mail** (`filter_osm_quality`; OSM liefert beides via `email`/`contact:email` + `website`/`contact:website`, jetzt beide extrahiert + `normalize_url`-normalisiert). **Google verlangt Website** (aus einem **Place-Details-Call** je Treffer, `GOOGLE_DETAILS_FIELDS=website,formatted_phone_number,business_status`) **und `business_status=OPERATIONAL`** (Text-Suche filtert Geschlossene schon vorab in `parse_google`). **E-Mail liefert Google Places grundsätzlich NIE** → Google-Leads haben keine E-Mail (nur Website+Telefon); „E-Mail-Pflicht" würde Google auf 0 setzen — daher bewusst nur Website-Pflicht bei Google. **Gemessene OSM-Abdeckung:** Website ~45–60 %, E-Mail ~16–23 %, **beides ~15–22 %** — Großstadt-Segmente behalten dutzende gute Leads, Kleinstädte/Nischen können dünn/leer sein. `discover_google` gibt `(candidates, api_calls)` zurück; der Router zählt den Nutzungszähler um die **echten** API-Calls hoch (`bump(..., increment=api_calls)` = 1 Text-Suche + N Details) — Place-Details sind einzeln billbar, das schlägt aufs Google-Kontingent.
  - Pipeline-Karten zeigen ein **Quelle-Badge** (`leadSources.ts::sourceBadge` → Label+Farbe: LinkedIn/OSM/Google/Manuell/Freitext) und die Pipeline hat **Quellen-Filter-Chips** (nach Häufigkeit, `sourceKey` fasst gleichartige zusammen).
- **Duplikat-Erkennung (zentral, `services/lead_dedup.py`, 2026-07):** Alle Import-Pfade (LinkedIn, OSM/Google-Discovery, manuelle Anlage, API) legen **keine Duplikate** an — weder gegen den Bestand noch batch-intern.
  - **Auto-Skip-Hierarchie:** E-Mail (normalisiert) → Website/Profil-URL (normalisiert, **Pfad bleibt** → LinkedIn-Profile distinkt) → exakter Ansprechpartner-Name. **Der Firmenname ist bewusst KEIN Auto-Key** — im Bestand sind z. B. 22 eigenständige Leads „Selbstständig" bzw. 10 bei „bayootec", je eigene Profil-URL; company-Matching würde sie fälschlich verschmelzen. Firma nur für Fuzzy-Warnung (`norm_company`, Rechtsformen GmbH/AG/e.K.… entfernt). Feldrealität (2355 Leads): Website 97,7 % · E-Mail 1,1 % · company 100 % (aber gemischt Firma/Personenname).
  - **DB-Absicherung (race-sicher):** zwei **generierte STORED-Spalten** `email_norm`/`website_norm` + **partielle Unique-Indizes** `(owner_id, email_norm/website_norm) WHERE … IS NOT NULL` (Migration `backend/scripts/add_lead_dedup_constraints.sql`, in Prod am 2026-07-13 eingespielt; 0 Bestandskonflikte). Die SQL-Normalisierung MUSS mit `lead_dedup.norm_email/norm_website` übereinstimmen. Inserts laufen über `_safe_flush_lead` (SAVEPOINT `begin_nested` + `IntegrityError`-Abfang → Skip statt 500).
  - **`DedupIndex`** (in-memory, pro Import einmal aus dem Bestand gebaut via `_build_dedup_index`): `match()` liefert `(lead, reason)`, nach jedem Insert `add()` → Batch-interne Duplikate. Manuelle Anlage `POST /rainmaker/leads` warnt mit **409** (Detail-Objekt `{message, reason, existing_id}`) statt hart zu blocken; `?force=true` legt trotzdem an (Frontend `LeadForm`: `window.confirm` → Retry mit force).
  - **Import-Report:** `LinkedInImportResult`/`LeadDiscoveryResult` tragen `skipped_rows:[{name, reason}]` (reason = email/website/name); Discovery-Preview-Kandidaten `duplicate_reason`. **Bestandsdaten-Report** (nur Report, kein Löschen): `python -m app.scripts.report_lead_duplicates` — Verdachtsgruppen pro Owner + Behalten-Vorschlag (ältester/vollständigster), Firmennamen-Kollisionen bewusst nur als Warnung.

## Key Gotchas
- **Pydantic `date` field shadowing**: If a model has a field named `date`, import as `from datetime import date as DateType` to avoid `date | None` failing in subsequent classes in the same file.
- **Docker .env `$` escaping**: bcrypt hashes in `ADMIN_PASSWORD_HASH` must escape `$` as `$$`.
- **page_size limit**: Set to `le=1000` on all list endpoints (frontend requests 1000 for dropdowns).
- **PDF footer**: Use `@page @bottom-center` in CSS, NOT `position: fixed` (causes overlap with content).
- **PDF Anschriftenblock**: the customer address field is single-line free text — `services/address_format.py::format_address_lines` splits it into DIN-5008-style lines for invoice/quote/reminder PDFs (comma-separated, comma-less via 5-digit-PLZ boundary, multiline passthrough; hyphens in streets/cities/number ranges stay intact; a trailing standalone "Deutschland" is dropped for domestic mail). Registered as Jinja filter `address_lines` on all three pdf_service Environments — new templates must use it instead of dumping `customer.address` raw.
- **Nginx timeout**: Set to 120s in `nginx/default.conf` for PageSpeed and other long-running API calls. Max body size 10M.
- **Nginx 502 after deploy**: Always restart nginx after rebuilding containers (new container IPs).
- **Signature in documents**: Image goes ABOVE the line (`border-top`), text BELOW. Signature HTML is a placeholder `{signature_html}` replaced during rendering.
- **Discount storage**: Stored as `discount_type` (percent/fixed), `discount_value`, `discount_reason` on Invoice model — NOT as negative positions. Guard: `discount_value is not None and discount_value != 0`.
- **Discount clear on PUT**: `update_invoice` distinguishes "field present in body" (`"discount_type" in update_data`) from `is not None` — the form sends explicit `null` to remove a discount, and a `None`-check would recompute totals with the OLD discount while the row is nulled (silent money mismatch, also leaked into the regenerated PDF). Fixed 2026-07.
- **Special terms**: Stored as JSON array string in `special_terms` field. Single string also supported (backward compatible).
- **Async httpx in refresh-drafts**: Uses `httpx.AsyncClient` (not sync `httpx.get`) to avoid blocking the event loop.
- **PDF generation in threads**: All `generate_*_pdf()` calls wrapped via `asyncio.to_thread()` since WeasyPrint is sync and would block the event loop for 2-5s per render.
- **PDF regen on invoice save is backgrounded**: `update_invoice` (PUT) used to regenerate the PDF synchronously in-request — WeasyPrint (2-5s) + token-tracker fetch (`timeout` now 8s, was 30s) + GitHub commit fetch (N+1) made saving take 4-8s. It now schedules `_regenerate_pdf_bg(invoice.id)` via FastAPI `BackgroundTasks` (own session); the PUT returns in ~130ms and the PDF updates a moment later (brief staleness window if the user downloads immediately). The explicit `POST /{id}/generate-pdf` stays synchronous (user waits for the file). Create never generated a PDF (already fast).
- **CORS_ORIGINS env var required**: Empty value blocks all cross-origin requests. Set to `https://ops.celox.io` (or local dev origin) in `.env`.
- **JWT_SECRET startup validation**: Server refuses to start if `JWT_SECRET` is the default `"change-me-in-production"` or shorter than 32 chars. Generate with `python -c "import secrets; print(secrets.token_urlsafe(48))"`.
- **Dashboard revenue chart** (`GET /api/dashboard/charts`, cached per `(owner, period, include_drafts)` in `_charts_cache`): `revenue_by_period` rows carry `invoice_count`, `customer_name` (only when exactly 1 invoice — via `min(Customer.name)` aggregate), `issued_count` (statuses gestellt/bezahlt/ueberfaellig by `invoice_date`, deliberately toggle-independent) and `paid_count`/`open_count`/`draft_count` (conditional `case()` aggregates). **`include_drafts=true` means EXPECTED revenue** = bezahlt + gestellt + ueberfaellig + entwurf (not just paid+drafts — issued-but-unpaid must not vanish). Frontend: tooltip strings live in `utils/chartTooltip.ts` (`revenueTooltipLabel` — testable, count/customer/status-breakdown logic); the dashed 'Rechnungen gestellt' bar uses a mini Chart.js plugin (`dashedBars` in Dashboard.tsx) because bar elements have no native `borderDash`, on a second right-hand count axis `y1`.
- **Stats cache**: `/api/dashboard/stats` cached in-memory for 60s (`_stats_cache` in `routers/dashboard.py`). **AuditMiddleware** calls `invalidate_stats_cache()` after every successful mutating `/api/` request (2xx/3xx, excluding `/api/dashboard/*`) — so mark-paid/edit/delete reflects on the dashboard instantly, not after 60s. Overdue-Cron invalidates too when it flips statuses. Container restart also clears the cache (in-memory).
- **FK reassignment on update is owner-validated**: `update_invoice`/`update_order`/`update_contract` validate `customer_id`/`order_id`/`contract_id` from the body with a scoped `select` before `setattr` — `with_loader_criteria` only hides rows on SELECT, and the post-update `joinedload(customer)` is a relationship load (unfiltered), so an unvalidated reassignment leaked a foreign tenant's name/address into the response AND the regenerated PDF. Fixed 2026-07.
- **Customer relationships are `lazy="raise"`**: Don't access `customer.orders/contracts/invoices` without explicit `joinedload()` in the query — will throw at runtime.
- **Attachment MIME whitelist**: Only PDF, images, Office, ZIP accepted. Other types → 415. Whitelist in `routers/attachments.py`.
- **Login rate limit**: 5/min/IP via `slowapi`. Decorator on `auth.login`, exception handler in `main.py`.
- **2FA (per-user)**: TOTP via `pyotp`, stored in `users.totp_secret` (nullable = off). Self-service in **Einstellungen**: `GET /api/auth/2fa/init` (fresh secret + QR data-URL, not persisted) → `POST /api/auth/2fa/enable` {secret, code} (verifies then persists) → `POST /api/auth/2fa/disable` {code}. When a user has `totp_secret`, login requires the 6-digit code in the OAuth2 `scope` field; Login.tsx shows the TOTP input on the first 401 "2FA-Code erforderlich". The legacy env `TOTP_SECRET` is only copied into the admin's `totp_secret` at bootstrap.
- **401 interceptor exception**: `api/client.ts` redirects to `/login` on 401 ONLY when the request is NOT to `/auth/login` AND the user is NOT already on `/login`. Otherwise the user gets bounced before seeing form errors (e.g. wrong password, 2FA required).

Precaches the shell (`offline.html`, manifest, icons) on install. Cache name `celox-ops-v6` — **bump `CACHE_VERSION` on any app-shell change** to purge old caches. **Stale-chunk protection (3 layers, 2026-07):** (1) `frontend/nginx.conf` serves `/assets/` with `try_files $uri =404` — a missing chunk (old hash after a deploy) returns 404, NOT the SPA-fallback `index.html` (which browsers/SW would cache under the .js URL → "Expected a JavaScript module … MIME type text/html"); (2) the SW content-type guard above; (3) `main.tsx` listens for Vite's `vite:preloadError` (failed dynamic import of a lazy route) and reloads the page once (sessionStorage-guarded against loops) so open tabs self-heal after a deploy. PWA icons: `public/icon-{192,512}.png` (`any`) + `icon-maskable-{192,512}.png` (`maskable`, generated from `icon-maskable.svg`) + `apple-touch-icon.png` (180, iOS — SVG isn't supported there); regenerate via ImageMagick (`magick -background "#0d1117" -density 900 favicon.svg -flatten -resize NxN out.png`). - **Audit log + cache invalidation**: Middleware in `app/middleware/audit.py` (1) logs all mutating requests to `audit_log` (best-effort — never blocks the response), (2) invalidates the dashboard stats cache after every successful mutating `/api/` request.
